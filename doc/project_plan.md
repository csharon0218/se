# 1. 서론

## 1.1 문서의 목적 및 범위

본 프로젝트 관리 계획서는 **SyncSpace(클라우드 파일 공유 시스템)** 프로젝트의 전반적인 내용을 정리하여 프로젝트를 진행하는 데 있어 각 세부 사항의 기준으로 삼는 것을 목적으로 한다.

본 문서는 다음과 같은 범위의 기준으로 적용한다.

① 프로젝트 개요, 프로젝트 정의, 주요 기능 정리  
② 개발 계획 및 환경, 개발 절차 모형, 개발 내용 및 일정 정리  
③ 팀 구성, 역할 및 책임 정리  
④ 품질 관리, 변경사항 및 산출물 관리 정리  
⑤ 산출물 정의, 작성일 및 담당자, 리스크 대응 기준 정리  

즉, 본 문서는 프로젝트의 기능 구현뿐 아니라 **문서 작성 절차, 기능 개발 흐름, 품질 관리 방식, 리스크 대응 기준**까지 포함하는 관리 문서이다.

## 1.2 프로젝트 개요

### 1.2.1 프로젝트 정의

SyncSpace는 조직 내부 파일을 안전하게 저장하고, 빠르게 탐색하며, 권한에 따라 정확하게 공유할 수 있도록 지원하는 **웹 기반 클라우드 파일 공유 시스템**이다.

현재 조직 내 문서, 설계 파일, 보고서, 이미지 등 다양한 업무 자료는 이메일, 메신저, 개인 저장소 등에 분산되어 관리될 수 있으며, 이로 인해 다음과 같은 문제가 발생할 수 있다.

- 파일 최신 버전 혼선
- 필요한 파일 탐색 시간 증가
- 외부 공유 시 보안 통제의 어려움

본 프로젝트는 이러한 문제를 줄이기 위해 파일의 중앙 집중 관리, 검색, 공유, 버전 관리, 권한 기반 접근 제어 기능을 제공하는 시스템 구축을 목표로 한다.

|---|---|
| F01 | 파일 업로드 및 다운로드 |

| F10 | 대용량 업무 파일 업로드 지원 |

## 1.3 프로젝트 목표

본 프로젝트의 목표는 다음과 같다.

첫째, 조직 내 파일을 한곳에서 관리할 수 있는 중앙 집중형 파일 관리 환경을 구축한다.  
둘째, 사용자가 파일을 쉽고 빠르게 업로드, 다운로드, 검색, 공유할 수 있도록 한다.  
셋째, 외부 공유 링크와 권한 기반 접근 제어를 통해 보안성을 강화한다.  
넷째, 삭제 복구와 버전 복원 기능을 통해 협업 과정의 실수를 회복 가능하게 한다.  
다섯째, 향후 기능 확장을 고려한 구조를 설계한다.

특히 본 프로젝트는 단순 저장 기능이 아니라 **버전 혼선 최소화, 파일 탐색 시간 단축, 외부 공유 시 보안 리스크 감소**를 실제 시스템 목표로 연결하는 데 의미가 있다.

# 2. 개발 계획

## 2.1 개발 절차 모형

### 1) 소프트웨어를 개발하기 위한 개발 절차 모형

본 프로젝트는 초기 기능 범위는 비교적 명확하지만, 문서 작성 과정에서 세부 요구사항과 우선순위가 점차 구체화될 수 있다. 또한 과제 특성상 단계별 산출물이 중요하고, 피드백 반영에 따라 세부 기준이 보완될 수 있으므로 **Agile 방법 중 SCRUM 기반의 반복적·점진적 개발 방식**을 적용한다.

즉, 전체적으로는  
**제안 → 품질 요소 도출 → 관리 계획 → 요구사항 정의 → 요구사항 분석 → 설계 → 구현 → 테스트**  
순으로 문서를 정교화하고, 구현 단계에서는 핵심 기능부터 우선적으로 개발하는 방식으로 진행한다.

본 계획에서 SCRUM 기반 방식을 채택한 이유는 다음과 같다.

- 과제 진행 중 문서 피드백에 따라 세부 기준이 보완될 수 있음
- 핵심 기능과 부가 기능의 우선순위를 구분하여 관리할 필요가 있음
- 검색, 공유, 버전 관리, 보안 기능은 상호 연관성이 높아 단계적으로 검토하는 것이 효율적임
- 품질 요소 추정서에서 정리한 성능, 보안성, 신뢰성 기준을 이후 요구사항과 설계에 반영해야 함

### 2) 개발 절차 모형과 함께 분석·설계를 진행하기 위한 방법론

분석과 설계는 기능별 책임을 분리하는 방향으로 진행한다.  
특히 SyncSpace는 파일 저장, 검색, 공유, 권한, 버전 관리, 대용량 업로드 처리가 핵심 기능이므로, 이후 설계 단계에서는 기능별 모듈 분리를 고려한 구조로 구체화한다.

이와 같은 방향은 유지보수성, 모듈성, 테스트 용이성, 확장성 확보와도 연결된다.

## 2.2 WBS(Work Breakdown Structure)

본 프로젝트의 WBS는 다음과 같이 정의한다.

### WBS 구조

**SyncSpace 소프트웨어 개발**
- 요구분석
  - 프로젝트 제안서 검토
  - 유사 시스템 조사
  - 요구사항 및 범위 정의
  - 필요 기능 정의
- 설계
  - 품질 요소 검토
  - 시스템 구조 설계
  - 데이터 및 권한 구조 설계
  - 상세 설계
- 구현
  - 기능 구현
  - 저장/검색/공유 기능 개발
  - 버전 관리 및 권한 기능 개발
  - 기능 통합
- 테스트
  - 전체 기능 테스트
  - 시나리오 기반 테스트
  - 문서 및 결과 정리
  - 발표 자료 작성

## 2.3 개발 내용

| 개발 활동 | 설명 |
|---|---|
| 프로젝트 요약 | 프로젝트 주제 선정 배경, 문제 정의, 목표 시스템 방향을 정리하고 프로젝트 전체 개요를 문서화한다. |
| 프로젝트 계획 | 프로젝트 관리 계획서를 작성하여 개발 절차, 일정, 리스크 관리, 산출물 관리 기준을 수립한다. |
| 요구사항 정의 | 기능적 요구사항, 비기능적 요구사항, 인터페이스 요구사항을 구체적으로 정리하여 요구사항 정의서를 작성한다. |
| 요구사항 분석 | 정의된 요구사항 간 관계와 우선순위를 분석하고, 구현 범위와 제약사항을 정리하여 요구사항 분석서를 작성한다. |
| Product Backlog | 핵심 기능과 부가 기능을 우선순위대로 정리하여 Sprint 수행의 기준이 되는 작업 목록을 구성한다. |
| Sprint Planning Meeting | 각 Sprint 시작 시 해당 기간의 목표 기능, 문서 작성 범위, 우선 수행 작업을 선정하고 계획한다. |
| Daily Scrum Meeting | 진행 상황, 이슈 사항, 다음 작업 계획을 일일 단위로 점검하여 일정 지연과 누락을 방지한다. |
| Sprint 1 | 프로젝트 제안서와 품질 요소 추정서를 검토하고, 요구사항 정의서 초안에 필요한 핵심 기능 목록을 정리한다. |
| Sprint 2 | 기능적 요구사항, 비기능적 요구사항, 인터페이스 요구사항을 구체화하여 요구사항 정의서를 작성한다. |
| Sprint 3 | 요구사항 간 관계와 우선순위를 분석하고, 구현 범위 및 핵심 기능 범위를 정리하여 요구사항 분석서를 작성한다. |
| Sprint 4 | 시스템 구조, 데이터 구조, 권한 관리 흐름, 버전 관리 흐름 등 설계 방향을 정리하여 설계 초안을 작성한다. |
| Sprint 5 | 파일 업로드/다운로드, 폴더 관리, 기본 파일 구조 등 핵심 기능 1차 구현을 진행하고 기본 동작 흐름을 점검한다. |
| Sprint 6 | 검색, 공유, 외부 링크, 권한 제어 등 핵심 기능 2차 구현을 진행하고 기능 간 연동을 검토한다. |
| Sprint 7 | 구현된 기능을 바탕으로 시나리오 기반 테스트를 수행하고, 품질 요소 추정서의 기준과 실제 결과의 정합성을 확인한다. |
| Sprint 8 | 전체 산출물과 구현 결과를 최종 보완하고, 피드백 반영, 발표 자료 작성, 제출 자료 정리를 수행한다. |
| Sprint Retrospective | 각 Sprint 종료 후 진행 과정을 돌아보고 잘한 점, 개선할 점, 다음 Sprint에 반영할 사항을 정리한다. |
| Sprint Review | 각 Sprint 종료 후 산출물과 수행 결과를 점검하고, 피드백을 반영하여 다음 단계 계획에 연결한다. |

※ 각 Sprint는 해당 주차의 문서 작성 또는 기능 구현 목표를 중심으로 수행하며, Sprint 종료 후 Review와 Retrospective를 통해 다음 단계 계획에 반영한다.

# 3. 팀 구성

## 3.1 팀 구조

본 프로젝트는 1인 수행을 기준으로 진행하며, 프로젝트 기획, 문서 작성, 요구사항 분석, 설계 검토, 구현, 테스트, 최종 정리까지 전 과정을 수행자가 통합적으로 관리하는 구조를 따른다.

따라서 별도의 역할 분담보다는, 프로젝트 전체 흐름을 일관되게 유지하고 각 산출물 간 정합성을 확보하는 것을 중요한 관리 원칙으로 삼는다.

## 3.2 역할 및 책임

| 이름 | 역할 | 책임 |
|---|---|---|
| 최민서 | 프로젝트 기획 및 문서 총괄 | 프로젝트 주제 설정, 제안서 작성, 품질 요소 추정서 작성, 프로젝트 관리 계획 수립, 요구사항 정의 및 분석, 설계 방향 정리, 핵심 기능 구현 및 점검, 문서 간 정합성 검토, 최종 제출본 통합 관리 |

# 4. 품질 관리

## 4.1 팀 미팅 계획

### (1) Sprint Planning Meeting
- 진행 시점 : 각 Sprint 시작 시점
- 토의 내용 : 해당 Sprint의 목표, 작성할 문서, 구현할 기능, 검토할 항목, 우선 수행 작업을 계획한다.

### (2) Daily Check
- 진행 시점 : 필요 시 일일 단위 점검
- 토의 내용 : 이전 작업 진행 상황, 현재 이슈, 다음 작업 계획을 점검한다.

### (3) Sprint Review
- 진행 시점 : 각 Sprint 종료 시점
- 토의 내용 : Sprint 기간 동안 수행한 작업과 산출물을 검토하고, 다음 단계에 반영할 사항을 정리한다.

### (4) Sprint Retrospective
- 진행 시점 : 각 Sprint 종료 시점
- 토의 내용 : 진행 과정에서 잘된 점, 부족한 점, 개선이 필요한 점을 정리하여 다음 Sprint 계획에 반영한다.

### (5) 비정기 검토
- 프로젝트 진행 중 문서 구조 수정, 피드백 반영, 일정 조정이 필요한 경우 수시로 검토를 진행한다.

## 4.2 변경사항 관리 방법

- 기능 변경이나 문서 구조 변경이 발생하는 경우 변경 전/후 내용을 반드시 기록한다.
- 변경 이유는 단순 수정이 아니라 어떤 피드백, 어떤 판단, 어떤 정합성 문제에 의해 변경되었는지 함께 남긴다.
- 제안서, 품질 요소 추정서, 프로젝트 관리 계획서, 요구사항 정의서 간 내용이 충돌하지 않도록 상호 검토한다.
- 핵심 기능 변경이나 품질 기준 변경은 이후 산출물에도 일괄 반영한다.
- 중요한 변경사항은 GitHub commit, PR, issue 답변 등 수정 이력이 남는 방식으로 관리한다.

## 4.3 산출물 관리 방법

### (1) 산출물에 대한 명명 방법
- 소프트웨어 : `v버전_산출일자`
  - 예) `v1_2026_04_20`
- 문서 : `[팀명]문서명_YYYYMMDD_Doc-001.md`
  - 예) `[SyncSpace]프로젝트관리계획서_20260420_Doc-001.md`
- 그 외 파일 : 목적이 명확히 드러나는 이름 사용

### (2) 산출물의 버전 관리 방법 및 버전관리 담당자
- 소프트웨어는 파일명 앞에 버전을 기재한다.
- 문서는 파일명 뒤에 날짜를 기재하고 문서 내 제/개정 이력을 수정한다.
- 큰 구조 수정이 있으면 버전을 올리고 변경사항을 기록한다.
- 버전 관리 담당자는 최민서로 한다.

### (3) 산출물의 저장 방법
- 소프트웨어 : GitHub 저장소
- 문서 : GitHub 저장소 `doc/` 디렉토리 및 로컬 백업
- 그 외 파일 : 로컬 및 클라우드 백업
